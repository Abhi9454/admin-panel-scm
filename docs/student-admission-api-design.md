# Student Admission — API Design Reference

> Generated by analysis of:
> - `src/views/student/AddStudent.js`
> - `src/views/student/AdditionalStudentDetails.js`
> - `src/views/student/EditStudent.js`
> - `src/api/studentManagementApi.js`
> - All interlinked lookup/dropdown sources

---

## 1. Overview — Two-Phase Admission Flow

The UI splits admission into **two sequential steps**:

```
Phase 1 — Core Admission (AddStudent.js)
  └─► POST /admin/students/
        └─► Returns adm_no + student_id on success
              └─► Phase 2 auto-renders (AdditionalStudentDetails.js)

Phase 2 — Extended Details (AdditionalStudentDetails.js)
  └─► PATCH /admin/students/{adm_no}/
        (parent info, address, medical, academic history)
```

The frontend waits for Phase 1 to succeed and receive `adm_no` before
showing Phase 2. Both phases hit the SAME student resource but at
different times with different payloads.

## 3. Phase 1 — Create Student (Core Admission)

### Endpoint
```
POST /admin/students/
Content-Type: multipart/form-data   ← if photo attached
Content-Type: application/json      ← if no photo
```

### Request Fields

| Field           | Type     | Required | Notes                                         |
|-----------------|----------|----------|-----------------------------------------------|
| `adm_no`        | string   | YES      | Admission number — unique per school          |
| `name`          | string   | YES      | Full name of student                          |
| `student_type`  | string   | YES      | `"new"` or `"old"`                            |
| `class_id`      | integer  | YES      | FK → classes                                  |
| `section_id`    | integer  | YES      | FK → sections                                 |
| `group_id`      | integer  | YES      | FK → groups                                   |
| `gender`        | string   | YES      | `"M"` / `"F"` / `"O"` (Male/Female/Other)    |
| `city_id`       | integer  | YES      | FK → cities                                   |
| `state_id`      | integer  | YES      | FK → states                                   |
| `hostel_id`     | integer  | NO       | FK → hostels (null if not applicable)         |
| `locality_id`   | integer  | NO       | FK → localities                               |
| `father_name`   | string   | NO       |                                               |
| `mother_name`   | string   | NO       |                                               |
| `dob`           | date     | NO       | ISO format `YYYY-MM-DD`                       |
| `blood_group`   | string   | NO       | `A+`, `A-`, `B+`, `B-`, `O+`, `O-`, `AB+`, `AB-` |
| `religion`      | string   | NO       |                                               |
| `caste`         | string   | NO       |                                               |
| `aadhaar_no`    | string   | NO       | 12-digit Aadhaar                              |
| `enquiry`       | boolean  | NO       | Default `false`                               |
| `photo`         | file     | NO       | Student passport photo (jpg/png)              |

> **Current frontend field names** (camelCase) vs **required API field names** (snake_case):
>
> | Form field (old)    | API field (new)   |
> |---------------------|-------------------|
> | `admissionNumber`   | `adm_no`          |
> | `classNameId`       | `class_id`        |
> | `sectionId`         | `section_id`      |
> | `groupId`           | `group_id`        |
> | `hostel`            | `hostel_id`       |
> | `localityId`        | `locality_id`     |
> | `cityId`            | `city_id`         |
> | `stateId`           | `state_id`        |
> | `dateOfBirth`       | `dob`             |
> | `aadhaarNumber`     | `aadhaar_no`      |
> | `studentType`       | `student_type`    |
> | `fatherName`        | `father_name`     |
> | `motherName`        | `mother_name`     |
> | `photoFile`         | `photo`           |
> | Gender: `"Male"`    | `"M"`             |
> | Gender: `"Female"`  | `"F"`             |
> | Gender: `"Other"`   | `"O"`             |

### Success Response — `201 Created`
```json
{
  "adm_no": "2024001",
  "id": 42,
  "name": "Ravi Kumar",
  "class_title": "10th",
  "section_title": "A",
  "std_status": "active"
}
```

> The frontend checks for `response.id` to determine success and then
> passes `adm_no` to Phase 2. Return BOTH `id` and `adm_no`.

### Error Responses
```json
// Duplicate admission number
{ "adm_no": ["student with this admission number already exists."] }

// Validation error
{ "class_id": ["This field is required."] }
```

## 7. Photo Upload Strategy

Three photos can be uploaded during admission:
1. **Student photo** — in Phase 1 (POST)
2. **Father photo** — in Phase 2 (PATCH)
3. **Mother photo** — in Phase 2 (PATCH)

**Recommended:** Accept `multipart/form-data` whenever files are included.
The frontend detects if a file is selected and switches to FormData accordingly.

```js
// Frontend logic (already in AddStudent.js)
if (photoFile) {
  const fd = new FormData()
  // append all text fields + photo file
  await studentManagementApi.create(fd)  // sends multipart
} else {
  await studentManagementApi.create(payload)  // sends JSON
}
```

**Backend must:** Accept both `application/json` (no photos) and
`multipart/form-data` (with photos) on the same endpoint.

## 8. Locality — New Master Endpoint Needed

The form includes a **Locality** dropdown that currently calls the old API
(`/school-management/locality/all`). This needs a new master endpoint:

```
GET  /admin/localities/
POST /admin/localities/
GET  /admin/localities/{id}/
PUT  /admin/localities/{id}/
DELETE /admin/localities/{id}/
```

Response shape (same as other masters):
```json
{ "count": 10, "results": [{ "id": 1, "title": "Civil Lines", "seq_order": 1 }] }
```
